# Claude Code 精神说明

> 这不是一份“Claude Code 泄露代码讲解”，也不是功能清单。
>
> 这份文档只回答一个问题：
>
> **Claude Code 这类系统真正相信什么，它的设计根基是什么，以及这些东西对 Vibe 有什么可迁移价值。**

---

## 1. 一句话结论

Claude Code 的根基不是：

- “模型已经像一个成熟程序员”
- “只要多给几个 agent、多写几个 prompt 就能稳定交付”

它真正的根基是：

**把一个有生产力、但不完全可靠的推理核心，纳入一个比它更可靠的工程运行时。**

也就是说，它信模型有用，但不信模型足够可靠；它不把模型当权威，而把系统当权威。

---

## 2. 它解决的核心问题不是“会不会写代码”

表面上看，Claude Code 是一个 AI 编程工具。

但从整体设计看，它解决的核心问题其实不是：

- 模型能不能输出一段代码
- 模型能不能做一次看起来聪明的回答

它真正解决的是：

**如何让一个不完全可靠的模型，在真实的软件工程环境里，持续地产生可控、可恢复、可审计的状态变化。**

换句话说，它的重点不是“回答”，而是“行动”。

它关心的是：

- 这一步有没有副作用
- 谁批准了这一步
- 出错以后怎么恢复
- 会话中断以后怎么继续
- 工具调用是否越权
- 这次失败是环境问题、契约问题、还是设计问题

所以它的本质不是一个聊天产品，而是一个运行时。

---

## 3. Claude Code 的五个根本前提

这套系统背后有五个非常稳定的前提。理解这五条，比理解某个文件重要得多。

### 3.1 模型不是权威，只是决策部件

Claude Code 并没有把模型设计成“最终权威”。  
模型更像一个高压缩决策器，可以很快给出：

- 下一步做什么
- 先看哪个文件
- 哪个修复方向更可能收敛

但系统并不默认相信它会一直做对。

所以模型的输出必须被包进：

- tool contract
- permission gating
- write scope
- incident / checkpoint
- verification

这个前提非常关键：

**它不是让模型主宰系统，而是让模型在系统允许的边界里行动。**

### 3.2 真相不在模型里，在环境里

Claude Code 的世界观不是“模型理解了项目，所以它知道该怎么做”。  
它更像在说：

**真相不在权重里，真相在文件、命令输出、git 状态、测试日志、运行环境里。**

这直接导致它的架构会不断要求模型：

- 读取真实文件
- 读取真实命令输出
- 读取真实失败证据
- 基于这些证据再做下一步决策

这是一种非常强的“外化认知”设计。

它不试图让模型一次性想明白，而是让模型不断和环境闭环。

### 3.3 软件工程是“受约束的持续行动”，不是“高质量回答”

Claude Code 对编程的理解，不是“给出一个聪明答案”。  
它对编程的理解更接近：

**长期、可回放、带副作用、受约束的状态修改。**

这就是为什么它会高度重视：

- session persistence
- replay / restore
- background session
- remote / bridge
- checkpoint
- permission
- audit trail

因为在它看来，真正的软件工程不是一轮问答，而是一段持续运行的工作流。

### 3.4 人类和 AI 不是同一种操作主体

Claude Code 很明确地区分：

- 人类操作
- AI 操作

人类负责：

- 下目标
- 授权
- 控制策略
- 审批关键副作用
- 拥有会话

AI 负责：

- 提出和执行候选动作
- 调用工具
- 产出中间结果
- 在规则内推进任务

这就是为什么它一定会有：

- commands / tools 分层
- ask-user / permission / approval
- tool contract

因为它根本不认为“人类命令”和“模型动作”是一回事。

### 3.5 会话不是上下文窗口，而是运行中的进程

Claude Code 没把一轮会话看成“prompt -> response”。  
它更像把整个会话看成一个在运行的工程进程。

所以才会有：

- session persistence
- replay / restore
- checkpoint
- remote / bridge
- background session
- subagent

这些设计的根子都一样：

**AI 编程不是一次性推理，而是跨时间、跨状态、可恢复的执行过程。**

---

## 4. 它的精神到底是什么

把上面的东西压缩一下，Claude Code 的精神可以概括成三句话。

### 4.1 第一层：怀疑模型，但不放弃模型

它不是技术悲观主义。  
它承认模型有生产力，而且已经足够有用。

但它也不做技术浪漫主义。  
它不相信模型天然会：

- 稳定判断何时该行动
- 稳定判断何时该停
- 稳定处理真实副作用
- 稳定恢复中断上下文
- 稳定从高层目标推到低风险执行计划

所以它采取的不是“放手让模型来”，而是：

**用制度驯化模型。**

### 4.2 第二层：系统比智能更重要

Claude Code 最厉害的地方，不是 prompt 写得多花，而是它始终在强调：

- 边界
- 权限
- 状态
- 证据
- 恢复
- 审计

这意味着它相信：

**真正让系统可靠的，不是更聪明的模型，而是更清楚的系统结构。**

这也是为什么它的大量复杂度都不在“让模型多想一点”，而在：

- 工具能力合同
- 运行时边界
- 会话状态
- 故障恢复
- 授权和责任归属

### 4.3 第三层：工程现实高于角色扮演

很多 agent 系统喜欢做“多角色模拟团队”。  
Claude Code 不是完全没有角色，但它并不把“像人类团队”当核心。

它更在意的是：

- 哪个动作是否合法
- 这个动作的作用域是什么
- 这个动作属于哪个执行上下文
- 失败后该回到哪里

所以它最终长成的不是“AI 团队模拟器”，而是：

**工程执行操作系统。**

## 5. 哪些设计是长期的，哪些是模型补丁

理解 Claude Code 时，最重要的一点是区分：

- 哪些设计是“即使模型变强也会保留”的
- 哪些设计是“当前模型不够可靠，所以不得不补”的

### 5.1 长期设计

这些东西即使底层模型大幅增强，也大概率会保留。

#### `commands / tools` 分层

原因不是模型弱，而是产品需要把：

- 人类控制面
- 模型执行面

分开管理。

这是操作主体分离的结果，不是临时补丁。

#### tool system

它的真正作用不是“让模型会 function calling”，而是：

- 限定能力边界
- 标记副作用
- 做权限和审计
- 做进度和错误语义

即使模型变强，tool contract 仍然会保留，因为真实副作用总要有法律边界。

#### QueryEngine 里的“会话协调器”部分

如果把 QueryEngine 理解成“调用模型的包装器”，那就看浅了。  
它真正长期保留的部分，是会话级协调能力：

- 消息历史
- session 状态
- usage
- agent context
- tool loop 协调
- 运行模式统一

这里面有一些补丁逻辑会淡化，但“会话事务协调”本身不会消失。

#### session persistence / background / remote / bridge / subagent

这些都不是为了补模型，而是为了满足真实产品场景：

- 长任务
- 异步执行
- 多入口
- 跨终端
- 多代理协作

模型越强，这些需求只会更强，不会更弱。

### 5.2 模型补丁

这些设计很大程度上是在给当前模型的局限擦屁股，将来可能会收缩。

#### classifier / ask-user / recheck

这些存在的根因是：

- 模型并不稳定知道什么时候该问
- 也不稳定知道什么时候风险过高

所以系统加了额外分类器、复检和 ask-user 机制。

如果模型将来能更稳定地表达自己的不确定性，这些外层机制会收缩。

#### 大量 replay / restore / compact / retry / fallback 细节

这里面有产品需求，但很多具体实现是被模型和上下文限制逼出来的。

如果模型：

- 上下文更强
- 状态保持更稳
- 工具使用更可靠

这些“修复性流程”会变薄。

#### prompt-driven workflow 对 tool-driven workflow 的补位

Claude Code 现在不少高层动作仍然靠 prompt 包装，而不是完全结构化。

这本质上说明：

- 现在很多高层任务还不够适合做成刚性的 deterministic 流程
- 需要模型在工具和策略之间兜一层自由规划

将来这层会收缩，但不会完全消失。

## 6. 对 Vibe 来说，真正该学的不是“抄功能”，而是抄世界观

Claude Code 最值得学的，不是某个 patch 或某套 prompt，而是它的世界观。

### 6.1 先定义对象，再定义角色

Vibe 过去更容易从角色出发：

- pm
- architect
- env_engineer
- implementation_lead

而 Claude Code 的强处在于，它的稳定主语不是“角色”，而是：

- 会话
- 工具
- 权限
- 工单
- 状态转换
- 审计对象

对 Vibe 来说，这意味着：

**以后先定义运行时对象，再决定需要哪些 agent。**

### 6.2 先定义边界，再让模型自由发挥

Claude Code 的很多力量来自它一开始就决定：

- 哪些行为不能发生
- 哪些副作用必须授权
- 哪些状态转换是合法的
- 哪些对象才是真相

而不是等系统跑乱了再补规则。

对 Vibe 来说，这意味着：

**不要再让 prompt 先决定系统长成什么样。**

### 6.3 先做运行时，再做人格化

Claude Code 的角色感其实没那么重。  
它的核心不是“多像一个工程团队”，而是“像一个工程执行系统”。

对 Vibe 来说，这意味着：

- 不要再优先追求 agent 更像工程师
- 先让系统更像运行时

### 6.4 先做失败语义，再做成功路径

Claude Code 大量复杂度都在处理：

- 权限拒绝
- 副作用风险
- scope mismatch
- session 恢复
- 工具失败
- 中断续跑

这说明它非常重视失败语义。

对 Vibe 来说，这意味着：

**以后任何大功能，都要先写“失败后怎么办”，再写“成功时怎么跑”。**

## 7. 如果把 Claude Code 的精神翻成几条工程戒律

下面这几条，可以直接当成 Vibe 后续演化的设计戒律。

### 戒律 1：模型永远不是边界，系统才是边界

不要指望模型自己知道不能做什么。  
边界必须落在：

- schema
- tool contract
- scope
- permission
- checkpoint

### 戒律 2：不要让自然语言总结成为真相

真相必须尽量落在：

- artifacts
- ledger
- structured pack
- execution work order
- incident

自然语言只能做摘要，不能做最终裁决。

### 戒律 3：不要把“会跑”误认为“可维护”

一个东西能跑，只能说明它当前还能动。  
软件工程要的是：

- 后面还改得动
- 出错还找得到
- 中断还能接得上
- 新人还能读得懂

### 戒律 4：不要让一个模块同时拥有太多权力

如果一个模块同时负责：

- 判断
- 执行
- 恢复
- 记录
- 改 scope
- 拼 prompt

那它迟早会变成新的 `orchestrator.py`。

### 戒律 5：先把失败对象化，再谈智能修复

不要让 fix-loop 永远停留在“看着报错再试一次”。  
先把失败变成：

- ErrorObject
- IncidentPack
- FixPlanPack
- ExecutionWorkOrder

再让模型在这些对象上决策。

## 8. 不要误读 Claude Code

Claude Code 很容易被误读成：

- “他们 prompt 更厉害”
- “他们 agent 更多”
- “他们资源更大所以功能更全”

这些都不是最核心的。

更准确的理解是：

### 8.1 它不是“相信模型已经够强”

它恰恰是在承认：

- 模型还不够稳
- 模型还不够可依赖
- 模型还不能天然承担工程责任

所以才会长出这么厚的运行时。

### 8.2 它不是“AI 团队扮演系统”

它最终的主角不是 agent，而是 runtime。

### 8.3 它不是“代码泄露说明价值在代码”

真正有价值的不是具体代码，而是：

- 它相信什么
- 它如何看待模型
- 它如何安排责任、边界和失败

这才是可迁移的部分。

---

## 9. 对 Vibe 的直接行动建议

如果以后你要继续演化 Vibe，这份文档最应该落成下面这些具体动作。

### 9.1 优先事项

1. 继续把 `orchestrator` 拆成规则层和调度层  
2. 让 `ExecutionWorkOrder` 成为更多阶段的一等公民  
3. 把 fix-loop 的失败语义进一步对象化  
4. 让 README / system-map / prompt-index 跟代码一起演化  
5. 用测试保护边界，而不是只保护功能

### 9.2 不该优先做的事

- 再增加更多角色
- 再堆更多提示词
- 继续往大函数里塞判断
- 用“模型更聪明了”来掩盖系统边界不清

### 9.3 每次大改之前要问的 5 个问题

1. 这次改动新增了什么运行时对象？  
2. 这个对象在哪个阶段产生、在哪个阶段消费？  
3. 谁有权修改它？  
4. 失败以后退回哪里？  
5. 如果模型判断错了，系统怎么兜住？

---

## 10. 最后一句话

Claude Code 最核心的精神，不是“让 AI 像人类工程师一样工作”，而是：

**承认 AI 还不是一个可靠的工程师，然后用一个更强的系统，把它变成一个可用的工程执行体。**

对 Vibe 来说，真正该学的不是它长了多少功能，而是这句话背后的秩序感。
